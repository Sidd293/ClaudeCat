01Vision & Concept

The idea in one paragraph, the problem it solves, and why the mental model matters.

The Idea

ClaudeCat is a desktop app where a pixel-art cat lives in the corner of your screen. By default, it's asleep. You click it, a text box appears, you type what you want built. The cat wakes up, opens an "office" UI, and summons specialist cats — an Architect, a Coder, a Designer, a Marketer — each a Claude Code instance with a tailored system prompt. They work together inside a sandboxed Docker container, producing a runnable project. You watch cats bustle around an office while your app is built.

The Technical Core

Underneath the whimsy is a multi-agent orchestration pattern:

  • A host-side orchestrator plans work and supervises workers.
  • Each worker (cat) is a Claude Code process spawned inside a shared project container with a specialist system prompt.
  • Workers communicate through structured handoff JSON files on disk — the orchestrator never reads worker reasoning, only final handoffs. This bounds token cost and keeps the pipeline inspectable.
  • Projects are sandboxed in Docker with CPU/memory caps so full-auto execution (--dangerously-skip-permissions) is safe.

Why the Cat

Multi-agent orchestration frameworks already exist (CrewAI, AutoGen, LangGraph). What differentiates ClaudeCat is feel: the delight of watching pixel cats do your work. The cat metaphor isn't decoration — it's the product. Preserve it everywhere.

Key Boundaries

Host ↔ Container
Orchestrator lives on host (needs Docker socket). Workers live in container (sandboxed). /workspace bind-mount is the only shared surface.
Orchestrator ↔ Worker
One-way: orchestrator launches worker, worker writes handoff to disk, orchestrator reads handoff. No back-channel. No stdout ingestion.
Worker ↔ Worker
Indirect, via /workspace. Later workers read earlier handoffs and produced files. Never direct IPC.
UI ↔ Orchestrator
Local IPC (Electron main/renderer). Events flow up (progress), commands flow down (start project, cancel, approve checkpoint).

03Project Lifecycle

The seven phases a project moves through, from user prompt to running app.

Phase 1 — Intake

User clicks the sleeping cat. Text box appears. User types a goal. Orchestrator produces a project brief (goal, stack inference, skills needed, step count, estimated cost, ambiguities) and shows it to the user for one-click approval. Nothing is spawned until the brief is accepted.

Phase 2 — Provisioning

Orchestrator creates ~/claudecat/projects/<id>/, spins up a container with that dir bind-mounted to /workspace, labels the container with claudecat.session=<id>, and writes .claudecat/manifest.json (the project's source of truth).

Phase 3 — Planning

Orchestrator decomposes the brief into a task DAG. Some tasks parallelize, some are serial (see Section 9: shared-file contention). The DAG lives in the manifest.

v0 simplification

The POC hardcodes a two-step serial plan: architect → coder. LLM-driven DAG planning is a v1 feature. Do not build it yet.

Phase 4 — Execution

Each task spawns a scoped Claude Code instance inside the container with: (a) the cat's system prompt loaded via --append-system-prompt, (b) the task description as the main prompt, (c) a clear "done" criterion, (d) resource + time limits. Each cat writes a handoff artifact when finished.

Phase 5 — Checkpoints

Most steps proceed automatically. User approval is required only when:

  • An action would cost real money (deploying, paid APIs).
  • An operation is irreversible against non-sandbox resources.
  • A cat explicitly flags open_questions.
  • The same task fails twice in a row.

Phase 6 — Delivery

Orchestrator runs the final project (docker compose up or the coder's run_command), maps the port, generates a preview URL, writes a README summarizing what was built. User sees: live preview, source code in the bind-mounted dir, task summary.

Phase 7 — Teardown / Persistence

Default: ephemeral runtime. Container stops after N minutes idle; code persists on disk; project can be resumed. User can toggle "keep alive" for active projects. Explicit "archive" snapshots everything and frees resources.

04Skills & Cats Model

A "cat" is a bundle. Inspectable, testable, composable. Eventually there will be a marketplace — not yet.

Bundle Structure

skills/
├── coder-cat/
│   ├── manifest.json       # metadata, version, deps
│   ├── system-prompt.md    # the cat's persona + rules
│   ├── capabilities.json   # accepts / produces contract
│   ├── tools.json          # tool permissions
│   ├── examples/            # few-shot good outputs
│   ├── tests/               # eval cases
│   └── assets/              # sprite, sounds, sprite sheet
├── architect-cat/
└── designer-cat/

Manifest Schema

{
  "name":            "backend-coder-cat",
  "version":         "1.2.0",
  "author":          "ClaudeCat core",
  "accepts":         ["api-spec.md", "db-schema.sql"],
  "produces":        ["src/**/*.js", "handoff.md"],
  "requires_skills": [],
  "permissions":     { "network": true, "fs_write": true },
  "cost_estimate":   "medium",
  "sandbox_level":   "container"
}

The accepts and produces fields form the composition contract. The orchestrator uses these to validate that task outputs match next-task inputs.

Three Installation Tiers

TierSourcev0?
Built-inShips in the repo under workers/. Architect, Coder.✅ yes
Local / customUser drops a folder in ~/claudecat/skills/. Auto-discovered at startup.Stubbed (directory scanning only; no loading)
MarketplaceSigned bundles from verified publishers, downloaded on demand.❌ not until built-ins prove the model

Versioning

Projects pin cat versions in their manifest.json (backend-coder-cat@1.2.0). Rebuilding later uses the same version unless the user explicitly upgrades. This is package-lock.json for prompts.

05Handoff Contracts

The discipline that keeps this system from becoming an unparseable mess. Every cat writes one of these. The orchestrator reads only these, never worker output.

Required Fields

FieldTypeRequiredPurpose
task_idstring✓matches the task in the DAG (e.g., "architect")
status"completed" \| "failed"✓binary — drives orchestrator branching
summarystring, ≤1 sentence✓human-readable one-liner
files_createdstring[]recommendedorchestrator verifies these exist + non-empty
files_modifiedstring[]—same verification
assumptions_madestring[]—surfaced to user if later step conflicts
open_questionsstring[]—non-empty ⇒ checkpoint to user
handoff_to_nextstring—one-sentence hint for next cat

Example — Architect Output

{
  "task_id":          "architect",
  "status":           "completed",
  "summary":          "Designed a minimal Express + SQLite todo API.",
  "files_created":    ["spec.md"],
  "stack":            ["node", "express", "sqlite"],
  "run_command":      "npm install && npm start",
  "port":             3000,
  "assumptions_made": ["JSON body parsing", "no auth for v0"],
  "open_questions":   [],
  "handoff_to_next":  "Coder implements per spec.md"
}

The Three Failure Modes

  1. Success — valid handoff, status: "completed". Trust it, move on.
  2. Known failure — valid handoff, status: "failed", populated open_questions or summary explains what broke. Orchestrator decides: retry / escalate / skip.
  3. Unknown failure — no handoff at all. Worker crashed, hit token limit, got stuck. Orchestrator falls back to: read last N lines of events/<task>.log, report to user.

The system prompt of every cat must mandate writing the handoff even on failure. Belt + suspenders: a post-exec shim that force-writes a stub handoff if the worker didn't.

Verification

After reading a handoff, the orchestrator runs cheap deterministic verification — not LLM-based:

  • Do files in files_created actually exist?
  • Are they non-trivially sized (>2 bytes)?
  • If the handoff claims "tests pass," run the tests.
  • If it claims "install succeeded," no verification needed — the next task will surface it.

Workers that lie get caught. Verification failures are equivalent to task failures.

P1 — Robustness Fixes (before anything else)

  • Add a post-exec handoff shim: if the worker exits without writing {task_id}.json, the orchestrator writes a stub handoff with status: "failed" and reason: "no_handoff" plus the last 200 lines of events/{task_id}.log. soon
  • Add retry logic: per-task max 2 retries. After the second failure, surface to user with a clear message. soon
  • Add budget enforcement: track cumulative tokens consumed per run, halt if CLAUDECAT_BUDGET_USD is exceeded (estimate via Claude Code's stream-json token counters). soon
  • Add --user $(id -u):$(id -g) to the container spec for Linux bind-mount permission correctness.
  • Verify docker exec timeout is actually killed at the container level (orphan claude process check).

  • Add workers/designer.md — produces HTML/CSS when project is a static site.
  • Add workers/tester.md — writes and runs smoke tests against coder output.
  • Move planner.js to a capability-aware planner: given a goal, pick which cats to run from available workers/.

P3 — Preview URL + Final Run

  • After coder handoff, orchestrator starts the app inside the container using run_command.
  • Map port (from architect handoff) to a host port.
  • Print preview URL (e.g., http://localhost:<mapped_port>) to the terminal.
  • Later: Traefik-based subdomain routing for <project-id>.claudecat.local.

  • Define skills/*/manifest.json schema (see Section 4).
  • Migrate existing workers/*.md into bundles under skills/architect-cat/, skills/coder-cat/.
  • Loader: scan skills/ at orchestrator startup, register available cats by name.
  • Manifest-driven composition: orchestrator checks accepts / produces compatibility when sequencing tasks.

P5 — Electron UI (new surface, separate milestone)

  • Bootstrap Electron shell with a transparent always-on-top window for the floating cat.
  • Pixel sprite cat with CSS sprite-sheet animation (start: sleep → wake → idle).
  • Click-to-open text input; sends goal to orchestrator via IPC.
  • "Office view" window: shows active cats, streams task:log events per cat.
  • State machine for cat animations (sleep · wake · thinking · working · celebrating · failed).

08Acceptance Criteria

How to know the POC is actually working. Don't move past v0 until all of these are green.

Criterion 1 — Happy path

A user with Docker running and a valid API key can execute node src/orchestrator.js "build me a hello world express server" and get a project that runs on first try.

Criterion 2 — Handoff discipline

For 5 consecutive runs, every cat writes a valid .claudecat/handoffs/<task>.json with status, summary, and files_created. Zero missing handoffs. This is the most important metric. If cats skip the handoff, the whole architecture breaks down.

Criterion 3 — Token discipline

The orchestrator's own LLM token usage is zero (v0 has no LLM planner). Worker token counts are logged but the orchestrator never reads worker stdout into its context.

Criterion 4 — Sandbox integrity

No worker can write outside /workspace or escape the container. Resource caps (2 GB / 2 CPU) are enforced. Wall-clock timeout (10 min default) kills runaway workers.

Criterion 5 — Cleanup

After a run (success or failure), no orphan containers remain. docker ps -a --filter label=claudecat.session returns empty (or only the current session).

Criterion 6 — Verifiable failure

When a worker fails (intentionally induce by giving it an impossible task), the orchestrator exits with a clear error message and the project directory contains debuggable artifacts (logs + any partial handoff).

09Known Gotchas

Traps documented in advance. Read these before debugging.

Claude Code Auth

Claude Code inside the container authenticates via ANTHROPIC_API_KEY env var — not via OAuth / Claude Pro subscription. Subscription auth lives in ~/.claude/ on the host and is not container-portable. Users must bring their own API key.

Interactive Prompts

Claude Code sometimes prompts interactively (tool permission, destructive op confirmation). In a subprocess these hang forever. Mitigations in use: --dangerously-skip-permissions flag + non-interactive -p mode. If Claude Code adds new interactive prompts in future versions, workers will hang until timeout. Pin the Claude Code version in the Dockerfile once you find a stable one.

Docker Socket Mounting (future)

v0 does not mount /var/run/docker.sock into the project container. That means projects can't run their own docker compose up (for Postgres, etc.). When adding this, consider tecnativa/docker-socket-proxy to whitelist only safe API calls — raw socket access is root-equivalent to the host.

Bind-Mount Permissions on Linux

Container runs as UID 1000. If the host user is a different UID, files written by the container will be owned by the wrong user on the host. Fix: pass --user $(id -u):$(id -g) to the container and chown /workspace inside the entrypoint.

Shared-File Contention

When two workers run in parallel and both want to edit package.json, last-write-wins corrupts the project. Orchestrator rule (v1+): declare "shared files" in the plan, serialize access to them, parallelize everything else. Root configs (package.json, docker-compose.yml) are always serial; feature files are fair game to parallelize.

Container Startup Latency

Cold-start of the workspace container is 3–10 seconds. Noticeable to users watching a sleeping cat. Later: pre-warm a pool of one idle container; grab it when a project starts; refill in the background.

Token Cost Surprise

An 8-worker project at 20 min each can burn $40. Always show a live token counter in the UI. Enforce CLAUDECAT_BUDGET_USD. Show the final spend after each run. Users tolerate expensive; they don't tolerate surprise.

11Roadmap After v0

Rough ordering for what comes next. Each milestone is independently demo-able.

Milestone 1 — Hardening (2–3 days)

All P1 tasks: handoff shim, retries, budget, permissions. Goal: 10 consecutive successful runs on varied prompts.

Milestone 2 — Skill Bundle Format (1 week)

All P4 tasks: manifests, loader, composition checks. Migrate existing cats to bundle format. Add 3rd and 4th cats.

Milestone 3 — Preview URLs + Multi-Container (1 week)

Traefik setup, Docker Compose inside project container, sibling-container pattern. Projects can now spin up Postgres.

Milestone 4 — Electron UI (2 weeks)

All P5 tasks. Floating cat, office view, animated states. This is when the project becomes shareable.

Milestone 5 — Marketplace + Signing (open-ended)

Web registry, publisher verification, eval scores, permissions-at-install. Blocking issue: abuse prevention.
